## Dependency notes

- **001 gates everything.** It rebases the 3 local commits onto `upstream/main`
  and replaces our inline macOS Keychain helpers with upstream's
  `macos_keychain.py` (which also renamed the Keychain service and added a
  migration). Building features on the pre-rebase base compounds the conflict.
- **002 before 003 before 005.** 003 adds logging *inside* the core that 002
  consolidates; 005 wraps that consolidated, logged `cswap --monitor` loop. Doing
  005 first would create a third copy of the monitor loop — the exact drift 002
  removes.
- **004 needs only 001** (it reuses the post-rebase `list_accounts` renderer) and
  can run in parallel with 002/003/005.

```
001 ──┬──> 002 ──> 003 ──> 005
      └──> 004
```

## Decisions taken (by the user, 2026-06-13)

- **Sync strategy: rebase** the 3 commits onto `upstream/main` for a linear fork
  history (not a merge commit).
- **Service scope: macOS launchd only** this round; `service.py` is structured so
  a Linux `systemd --user` backend can be added later without CLI changes.
- **Code-style alignment**: the `cswap service` subcommand mirrors upstream's
  existing `cswap run` pre-dispatch pattern in `cli.py` (consistency / SSOT).

## Deferred (recorded so they aren't re-audited as gaps)

- **macOS Keychain read-timeout guard.** Our pre-rebase branch wrapped the
  `security` read with a 3s timeout (`_KEYCHAIN_READ_TIMEOUT_SECONDS` /
  `_KeychainReadTimeout`). Plan 001 drops it to keep the rebase mechanical;
  re-introduce it as a one-line wrapper around `macos_keychain.get_password`
  **only if** macOS users report hangs on a locked Keychain. Not worth doing
  pre-emptively.
- **Sister-test ThreadPoolExecutor flake.** Plan 001 deflaked
  `test_list_shows_rate_limit_when_no_previous_usage` by routing the mock
  with `side_effect=lambda` instead of an ordered list. The same root cause
  (nondeterministic `ThreadPoolExecutor.map` worker order) affects at least
  `test_list_preserves_previous_cached_usage_when_fetch_returns_none` and
  `test_list_shows_cached_usage_with_rate_limit_note` in
  `tests/test_switcher.py`. Stable on most runs; trip ~1 in 5 in isolation,
  rare in the full suite. Trivial follow-up: apply the same `side_effect=lambda`
  routing to those sister tests. Out of plan 001's scope; should land before
  CI starts running.

- **In-curses auto-refreshing usage dashboard.** Plan 004 reuses the CLI
  renderer (SSOT). A native curses table that live-refreshes all accounts would
  fork the formatting and needs a network-cadence decision — separate plan if
  ever wanted.
- **Linux/systemd + Windows service backends.** Out of scope per the platform
  decision; the launchd module is shaped to accept them later.

## Findings considered and rejected

- *Re-implement usage/health formatting natively in the TUI* — rejected: it
  would duplicate `switcher.list_accounts`, the opposite of the SSOT goal. Plan
  004 shells out to the one renderer instead.
- *Move threshold/usage logic out of `switcher` into the monitor core* —
  rejected: `switcher` is already the correct SSOT for account state; the core
  should call it, not absorb it (see plan 002 scope).
